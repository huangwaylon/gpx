# Internationalization (i18n)

How **Ume-chan's Trails** (梅ちゃんのトレイル) handles its two languages. The UI is
**Japanese by default** with a one-tap toggle to English. Everything user-facing is
translated: UI chrome, trail names/areas/summaries/descriptions/tips/permits, map marker
labels, difficulty/route/dog enums, seasons, times, and measurement units.

All translation logic lives in **`i18n.js`** (`window.I18N`) and a thin layer of helpers in
**`app.js`**. There is no i18n library and no build step — just plain objects and functions.

---

## 1. Mental model

There are two sources of text:

1. **`trails.js`** — `window.TRAILS`, the **English base** content (names, descriptions,
   tips, etc.) plus language-neutral data (slug, coords, stats, image/gpx paths).
2. **`i18n.js`** — `window.I18N`, holding **all UI strings** and the **Japanese overrides**
   for each trail, keyed by slug.

At render time the app picks strings for the active language. For trail bodies it *merges*
the Japanese block over the English base (`loc(trail)`), so any field not translated falls
back to English automatically.

```
TRAILS[slug]  (English base)  ─┐
                               ├─ loc(trail) ─► object used for rendering
I18N.trails[slug].ja  ─────────┘   (JA fields override when lang==='ja')
```

---

## 2. `window.I18N` structure (`i18n.js`)

| Key | Shape | Purpose |
|---|---|---|
| `ui.en` / `ui.ja` | `{ key: string }` | Static UI strings (buttons, headings, labels, alerts) |
| `fn.en` / `fn.ja` | `{ key: (…args) => string }` | Dynamic strings needing interpolation (download progress, etc.) |
| `diff` | `{ en:{}, ja:{} }` | Difficulty enum: `Easy/Moderate/Hard/Very Hard` → localized |
| `route` | `{ en:{}, ja:{} }` | Route enum: `Out & back/Loop/Point to point` → localized |
| `dogs` | `{ en:{}, ja:{} }` | Dog policy enum: `Leashed/Not allowed` → localized |
| `months` | `{ Jan:"1月", … }` | Month abbreviations for season ranges (JA only) |
| `wpt` | `{ "Bridge":"橋", … }` | GPX waypoint name → Japanese |
| `trails` | `{ "<slug>": { ja: {…} } }` | Per-trail Japanese content (see §4) |

### Why functions for some strings (`fn`)

Strings that embed numbers or names can't be naively concatenated across languages (word
order, particles, counters differ). They're stored as functions:

```js
fn: {
  en: { dlDesc: (name, n) => `Save ~${n} map tiles for "${name}" …` },
  ja: { dlDesc: (name, n) => `「${name}」の地図タイル（約${n}枚）を保存して…` },
}
```

Note the Japanese counter 枚 (the counter for flat objects like map tiles) — this is the kind
of nuance that a simple `"%s tiles"` template can't express.

## 3. The helper layer (`app.js`)

```js
let lang = (localStorage.lang === 'en' || localStorage.lang === 'ja')
  ? localStorage.lang : 'ja';        // Japanese is the default

const t  = key => I18N.ui[lang][key] ?? I18N.ui.en[key] ?? key;  // static string
const tf = key => I18N.fn[lang][key] ?? I18N.fn.en[key];          // dynamic string fn
```

| Helper | Returns |
|---|---|
| `t(key)` | A static UI string in the active language (falls back to EN, then the key) |
| `tf(key)` | A dynamic-string **function**; call it with args: `tf('dlDesc')(name, n)` |
| `loc(trail)` | The trail object with Japanese fields merged in when `lang==='ja'` |
| `trDiff(d)` | Localized difficulty (also used for the colored badge label) |
| `trRoute(r)` | Localized route type |
| `trDogs(g)` | Localized dog policy |
| `trSeason(s)` | `"Apr – Nov"` → `"4月～11月"` in JA (regex month + dash replacement); unchanged in EN |
| `trWpt(name)` | Localized GPX waypoint name (JA lookup, else the original) |
| `fmtDist(mi)` | `"9.8 km"` (JA) / `"6.1 mi"` (EN) |
| `fmtGain(ft)` | `"444 m"` (JA) / `"1,456 ft"` (EN) |
| `fmtTime(s)` | `"3時間17分"` (JA) / `"3h17min"` (EN) |
| `fmtElevRange(loM, hiM)` | profile axis label, meters (JA) / feet (EN) |

> ⚠️ **Naming caution:** the trail-localization helper is `loc()`, **not** `L()`. A single
> letter `L` is Leaflet's global; an earlier `function L(trail)` shadowed it and broke the
> map with `L.polyline is not a function`. Never reintroduce a global `L`.

### Units are part of i18n

Japanese hikers expect **metric**; the source stats are imperial (from AllTrails). The `fmt*`
helpers convert at render time — the stored data stays imperial (`lengthMi`, `gainFt`), and
only the *display* changes. Conversions: km = mi × 1.609344, m = ft ÷ 3.28084.

## 4. Per-trail Japanese content

Each trail has a Japanese block in `I18N.trails`, keyed by the same `slug` as in `trails.js`:

```js
I18N.trails = {
  "lake-22": { ja: {
    name:        "レイク22 トレイル",
    area:        "グラニットフォールズ（ワシントン州）",
    summary:     "原生林を抜けて高山湖へと至る美しいコース。…",
    description: "レイク・トゥエンティトゥー・トレイルは…",
    permit:      "ノースウエスト・フォレストパス、または日帰り利用料（Recreation.gov アプリ）",
    tips: [ "岩の多い道です。しっかりした登山靴が必須です。", … ],
  }},
  …
};
```

Only these text fields are translated; everything else (stats, coords, rating, image/gpx
paths, and the enum *values* like `"Moderate"`) stays in `trails.js` and is localized through
the enum tables. If a JA field is omitted, `loc()` falls back to the English value.

### Translation conventions used

- **Trail/place names** use katakana transliteration (レイク22 トレイル, スノーレイク) — this
  matches how Japanese hikers refer to foreign trails, rather than translating "Lake".
- **Prose** is natural Japanese, not literal — rewritten for flow, with metric distances woven
  into the text (e.g. "約1.6km地点で" for "about a mile in").
- **Polite register** (です・ます調) throughout, as is conventional for guide/UI copy.
- **Counters & units**: 枚 for tiles, km/m for distance/elevation, 時間/分 for time.

## 5. How rendering applies language

```
applyStaticI18n()   fills every [data-i18n] / [data-i18n-aria] node + document.title,
                    and sets <html lang>.
renderList()        builds cards using loc(trail), trDiff/trRoute, fmtDist/fmtGain.
renderPeek()        the bottom-sheet header (name + meta line).
renderSheetBody()   stat boxes, elevation card, sections, facts table, attribution.
drawTrack()         binds marker popups via t('markerTrailhead'/'markerEnd') and trWpt(name).
```

### Live language switching (no reload)

`setLang(next)`:

1. stores `localStorage.lang` and updates `lang`,
2. calls `applyStaticI18n()` and `renderList()`,
3. if a detail view is open, re-renders the title, peek, and sheet body, and
4. calls `redrawTrailLabels()` to re-bind the Leaflet marker popups (trailhead/end + waypoints)
   in the new language — the map itself is **not** rebuilt, so there's no flicker.

Static HTML carries the Japanese text inline as the default (so the first paint is correct
before JS runs) plus a `data-i18n` key so `applyStaticI18n()` can swap it:

```html
<h1 data-i18n="appName">梅ちゃんのトレイル</h1>
<button id="lang-toggle" class="lang-btn" data-i18n="langOther">EN</button>
```

The toggle button's own label is itself a translated string (`langOther`): it shows **EN**
while in Japanese and **日本語** while in English — i.e. it always names the *other* language.

## 6. Adding to the translations

**A new UI string:** add the same key to `I18N.ui.en` and `I18N.ui.ja`. If it interpolates
values, add it to `I18N.fn.en`/`I18N.fn.ja` as a function instead. Use it via `t('key')` /
`tf('key')(args)`, or place it in HTML as `data-i18n="key"`.

**A new trail:** add the English object to `trails.js`, then a `"<slug>": { ja: {…} }` block
to `I18N.trails` with `name, area, summary, description, permit, tips`. Translate any new GPX
waypoint names by adding entries to `I18N.wpt`.

**A new language (e.g. `ko`):** add `ui.ko`, `fn.ko`, and the `ko` side of each enum table
(`diff/route/dogs`); add a `ko` block per trail in `I18N.trails`; extend the `lang` validation
and the toggle in `app.js` (the current toggle is a simple JA↔EN flip — a 3rd language would
want a small menu). Units are currently chosen by `lang==='ja'` → metric, else imperial; a new
locale needs its own unit rule.

---

## 7. Persistence & first-run

- The chosen language is saved to `localStorage.lang` and restored on next load.
- With no stored value, the app starts in **Japanese**.
- `<html lang="ja">` is the static default; `applyStaticI18n()` updates `document.documentElement.lang`
  on every language change for correct text rendering and accessibility.
